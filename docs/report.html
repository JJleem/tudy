1. 도입 — 현직 개발자의 학습 딜레마

필자는 현직 소프트웨어 개발자로서 HYCU 사이버대학교 컴퓨터공학과에 재학 중이다. 업무상 새로운 기술을 빠르게 습득해야 하며, AI 도구 또한 일상적으로 활용한다. 그러나 기말고사 시즌에는 다음과 같은 경험이 반복되었다.

"강의도 확인하였고, ChatGPT·Claude에게도 질문하였는데 — 막상 시험 전날 개념을 설명하려니 아무 말도 나오지 않았다."

이에 대한 원인을 분석하였다. 기존 AI 활용 방식은 다음과 같았다: "퀵 정렬이 무엇인가?" → AI가 친절하게 설명 → 읽으면 이해된 것처럼 느껴짐 → 며칠 후 기억에 남은 내용이 없었다. 이는 수동적 학습(Passive Learning)의 전형적인 한계이다. 듣고 읽는 행위만으로는 장기 기억이 형성되지 않는다.

반면, 개념을 타인에게 설명하고자 할 때 학습자는 자신의 이해 범위와 한계를 명확히 인식하게 된다. 심리학에서는 이를 Protégé Effect(제자 효과)라 부른다 — 가르치기 위해 준비하는 과정이 오히려 본인의 이해를 심화시킨다.

기존 AI 학습 도구의 문제점

유튜브·강의 영상시청자이해한 것처럼 느껴지지만 직접 설명 불가

2. 아이디어 — 역할 반전 학습

핵심 아이디어는 간단하다: "AI가 설명하는 것이 아니라, 학습자가 AI에게 설명한다."

소크라테스 문답법(Socratic Method)에서 영감을 얻었다. 소크라테스는 지식을 직접 전달하지 않았다 — 끊임없는 질문으로 상대가 스스로 답을 찾게 하였다. AI는 이 역할에 최적화되어 있다. 지치지 않고 판단하지 않으며, 학습자의 답변 수준에 맞게 정확한 질문을 제시한다.

핵심 컨셉: 학습자가 개념 설명 → AI가 맞은 부분 인정·빠진 부분 질문 → 학습자가 다시 설명 → 반복 → 이해도 100점 달성
개념 선택
→

→

기존 도구와의 결정적 차이는 학습자의 역할이다. AI's TUDY에서 AI는 절대 먼저 설명하지 않는다. 학습자가 먼저 설명해야 하며, AI는 어느 부분이 맞고 어느 부분이 누락되었는지를 질문의 형태로 안내한다. 학습자는 설명을 반복하며 스스로 개념을 완성한다.

3. 개발 과정 — 바이브코딩으로 3일 만에

핵심 도전 1 — 소크라테스 프롬프트 설계

가장 난이도가 높은 부분이었다. 단순한 지시만으로는 AI가 직접 설명을 시작하는 문제가 발생하였다. 반복적인 시도 끝에 턴 기반 단계별 지침 주입 방식을 완성하였다. 각 턴마다 AI의 동작 지침을 시스템 프롬프트에 동적으로 추가하는 방식으로 해결하였다: 

## 절대 규칙
- 개념을 먼저 설명하지 마세요. 학생이 설명하게 만드세요.
- 한 번에 질문은 하나만 하세요. 응답은 6문장 이내.
- 반드시 한국어로만 대화하세요.

## 턴별 동작 (lib/prompts.ts → getPhaseGuide(turn))
- 1턴  : feedback-good / feedback-bad 블록 필수 — 초기 평가
- 2~3턴: 아직 다루지 않은 체크포인트 중심 피드백
- 4턴  : 심화 질문 (실제 코드 / 엣지케이스)
- 5턴  : 중간 정리 + score-box(이해도 점수) 필수
- 6턴+ : 매 턴 점수 업데이트, 100점 전 종료 금지

핵심 도전 2 — Claude API SSE 스트리밍 구현

응답이 실시간으로 화면에 타이핑되도록 SSE 방식으로 구현하였다. Next.js API Route에서 Claude 스트림을 그대로 클라이언트로 전달한다: 

// app/api/chat/route.ts
const stream = await anthropic.messages.stream({
  model: 'claude-sonnet-4-6',
  max_tokens: 1024,
  system: getSocratesSystemPrompt(concept, courseName, turn),  // 턴 주입
  messages,
})

const encoder = new TextEncoder()
const readable = new ReadableStream({
  async start(controller) {
    for await (const chunk of stream) {
      if (chunk.type === 'content_block_delta') {
        controller.enqueue(encoder.encode(
          `data: ${JSON.stringify({ text: chunk.delta.text })}\n\n`
        ))
      }
    }
    controller.enqueue(encoder.encode('data: [DONE]\n\n'))
    controller.close()
  },
})
return new Response(readable, { headers: { 'Content-Type': 'text/event-stream' } })

핵심 도전 3 — 피드백 UI 구현 및 렌더링 문제 해결

AI 응답 안에 HTML 블록이 포함되어 화면에서 색상 구분 피드백으로 렌더링된다. 일반 마크다운 파서(react-markdown)가 HTML 블록 내부의 **굵게**를 처리하지 않는 문제를 발견하였다. rehype-raw 플러그인 추가 후에도 동일한 문제가 발생하였다. 이에 전처리 함수를 작성하여 div 내부의 마크다운을 HTML 태그로 변환하는 방식으로 해결하였다: 

// components/SocratesChat.tsx
function preprocessMarkdownInHtml(content: string): string {
  return content.replace(
    /(<div[^>]*>)([\s\S]*?)(<\/div>)/g,
    (_, open, inner, close) => {
      const processed = inner
        .replace(/\*\*(.+?)\*\*/g, '<strong>$1</strong>')
        .replace(/\*(.+?)\*/g, '<em>$1</em>')
      return open + processed + close
    }
  )
}

AI 응답에서 아래와 같이 색상 구분 피드백이 렌더링된다(실제 앱 화면 기준):

✅ 이해한 부분 — 초록 블록으로 표시
❌ 빠진/틀린 부분 — 빨강 블록으로 표시

4.1 알고리즘실무 — 동적 프로그래밍(DP)

기말고사 범위 중 이해도가 가장 낮았던 동적 프로그래밍을 첫 번째 세션으로 선택하였다. 강의에서 '최적 부분 구조', '중복 부분 문제'라는 용어는 인지하고 있었으나, 실제로 설명하고자 할 때 명확한 표현이 어려웠다.

AI's TUDY의 DP 개념 카드를 선택하면 AI가 다음과 같이 묻는다: "동적 프로그래밍이 무엇인지 본인의 말로 설명해보세요." 학습자가 아는 내용부터 설명하자 AI는 정확한 부분(✅ 메모이제이션 언급)과 누락된 부분(❌ 최적 부분 구조 미설명)을 분리하여 제시하고, 직접 설명하는 대신 "왜 피보나치 수열이 DP의 전형적인 예인지 설명해보세요"로 유도하였다.

5.3 기술적 성과

개발 과정 자체도 중요한 학습 경험이었다. Claude API의 SSE 스트리밍을 최초로 구현하였고, HTML과 마크다운이 혼합된 콘텐츠를 렌더링하는 문제를 직접 해결하였다. rehype-raw가 HTML 내부 마크다운을 처리하지 않는 한계를 발견하고, 50줄 미만의 전처리 함수로 해결하였다.

아이디어 구체화에서 Vercel 배포까지 약 3일이 소요되었으며, 바이브코딩 방식이 아니었다면 훨씬 더 많은 시간이 필요했을 것으로 판단된다.

  • 학습 분석 — 여러 세션의 이해도 점수 추이를 그래프로 시각화하여 취약 개념 파악

6.3 결론

AI 학습 도구의 역설은 AI가 지나치게 잘 설명한다는 데 있다. 친절한 AI의 설명은 학습자를 수동적 수용자로 전락시킨다. AI's TUDY는 이 구조를 전환하여, AI를 학습자가 스스로 설명하도록 유도하는 도구로 활용하였다. 소크라테스가 2400년 전 발견한 진실 — "가르치는 자가 배운다" — 을 Claude API와 바이브코딩으로 구현한 실험이었다.
